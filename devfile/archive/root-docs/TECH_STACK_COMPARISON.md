# 技术栈对比分析

## 数据库选型对比

### MySQL vs MongoDB

| 维度 | MySQL | MongoDB |
|------|-------|---------|
| **数据模型** | 关系型（表格结构） | 文档型（JSON/BSON） |
| **查询语言** | SQL（标准化） | MongoDB Query Language（类JSON） |
| **Text2SQL适用性** | ⭐⭐⭐⭐⭐ 完美匹配 | ⭐ 不适用 |
| **企业采用率** | ⭐⭐⭐⭐⭐ 广泛使用 | ⭐⭐⭐ 特定场景 |
| **求职价值** | ⭐⭐⭐⭐⭐ 必备技能 | ⭐⭐⭐ 加分项 |
| **学习成本** | ⭐⭐ 低（SQL通用） | ⭐⭐⭐ 中等 |

**结论**: Text2SQL 项目必须选择 **MySQL**，因为：
- Text2SQL 的核心是将自然语言转换为 **SQL**
- MongoDB 使用非SQL查询语言，与项目目标不符
- MySQL 是企业级关系数据库的事实标准

### 最终数据库选型
- **SQLite**: 内置 Demo（轻量级，无需配置）
- **MySQL**: 主要生产数据库（企业标准）
- **PostgreSQL**: 备选方案（高级功能，但增加复杂度）

> **排除 MongoDB**: 与 Text2SQL 的 SQL 生成目标根本冲突

## LLM 集成方案对比

### Vanna AI vs 其他 NL2SQL 方案

| 方案 | 优势 | 劣势 | 适用性 |
|------|------|------|--------|
| **Vanna AI** | - 专为 NL2SQL 设计<br>- 内置训练机制<br>- 支持多数据库<br>- 开源免费 | - 相对新<br>- 社区较小 | ⭐⭐⭐⭐⭐ 完美匹配 |
| **LangChain SQL Agent** | - 生态成熟<br>- 插件丰富 | - 配置复杂<br>- 性能开销大 | ⭐⭐⭐ 可用但重 |
| **Custom LLM Prompt** | - 完全可控<br>- 轻量级 | - 需要大量提示工程<br>- 准确率不稳定 | ⭐⭐ 不推荐 |
| **Dify Built-in** | - 与 Dify 深度集成 | - 灵活性受限<br>- 功能有限 | ⭐⭐⭐ 备选 |

**Vanna AI 核心优势**:
1. **专为 NL2SQL 设计**: 不是通用 Agent，而是专门解决 NL2SQL 问题
2. **内置 RAG 机制**: 可以基于数据库 schema 进行上下文学习
3. **多数据库支持**: 原生支持 MySQL、PostgreSQL、SQLite 等
4. **开源免费**: MIT 许可证，无商业限制
5. **Python 原生**: 与项目技术栈完美匹配

### LLM Provider 对比

| Provider | 优势 | 劣势 | 成本 |
|----------|------|------|------|
| **Bailian Code Plan** | - 专为代码生成优化<br>- 中文支持好<br>- 阿里云生态 | - 新服务<br>- 文档较少 | 中等 |
| **OpenAI GPT** | - 成熟稳定<br>- 社区支持好 | - 国际访问<br>- 成本较高 | 高 |
| **Claude** | - 推理能力强<br>- 上下文长 | - 国际访问<br>- 中文一般 | 高 |
| **本地开源模型** | - 完全控制<br>- 无API成本 | - 需要GPU<br>- 准确率低 | 低（硬件） |

**选择 Bailian Code Plan 的理由**:
- 面向中国大陆用户（CloudBase部署约束）
- 专为代码/SQL生成优化
- 与阿里云生态集成良好

| 框架 | 开发速度 | 学习曲线 | 生产适用性 | 求职价值 |
|------|----------|----------|------------|----------|
| **Streamlit** | ⭐⭐⭐⭐⭐ 极快 | ⭐⭐⭐ 简单 | ⭐⭐⭐ Demo友好 | ⭐⭐⭐⭐ 热门 |
| **Gradio** | ⭐⭐⭐⭐ 快 | ⭐⭐⭐ 简单 | ⭐⭐⭐ Demo友好 | ⭐⭐⭐ 流行 |
| **Flask/Django** | ⭐⭐ 慢 | ⭐⭐⭐⭐ 复杂 | ⭐⭐⭐⭐⭐ 生产级 | ⭐⭐⭐⭐⭐ 必备 |

## 安全校验方案

### 自研 vs 第三方

| 方案 | 控制力 | 开发成本 | 可靠性 | 适用性 |
|------|--------|----------|--------|--------|
| **自研校验层** | ⭐⭐⭐⭐⭐ 完全控制 | ⭐⭐⭐ 中等 | ⭐⭐⭐⭐ 可控 | ⭐⭐⭐⭐⭐ 定制化 |
| **SQLAlchemy 安全模式** | ⭐⭐⭐ 部分控制 | ⭐ 低 | ⭐⭐⭐⭐ 成熟 | ⭐⭐⭐ 通用 |
| **第三方安全库** | ⭐ 无控制 | ⭐ 低 | ⭐⭐⭐ 依赖外部 | ⭐⭐ 有限 |

**选择自研安全校验层的理由**:
- Text2SQL 场景特殊，需要定制化安全规则
- 可以实现更精细的权限控制（表级别、字段级别）
- 符合企业级安全要求

## 最终技术栈决策

### 核心原则
1. **求职导向**: 选择热门、实用的技术栈
2. **开发效率**: 平衡快速开发和生产质量  
3. **企业标准**: 符合实际生产环境要求
4. **项目匹配**: 技术栈必须服务于 Text2SQL 核心目标

### 最终选型
- **数据库**: MySQL (主) + SQLite (Demo)
- **LLM**: Vanna AI + Bailian Code Plan
- **Web**: Streamlit (UI) + FastAPI (API)
- **安全**: 自研校验层 + SQLAlchemy
- **部署**: Docker + CloudBase

此技术栈组合在 **求职价值**、**开发效率**、**企业适用性** 三个维度达到最佳平衡。
