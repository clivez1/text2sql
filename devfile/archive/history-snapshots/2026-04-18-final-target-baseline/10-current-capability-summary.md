# Text2SQL 当前能力轻型总结

## 1. 项目定位

当前项目是一个面向演示与原型验证的 Text2SQL 系统，目标是把中文问题转成安全可执行的查询，并返回结构化结果与可视化建议。

更准确地说，它现在是：
- 一个可运行的 BI 查询 Demo
- 一个本地规则优先、LLM 兜底补充的查询系统
- 一个支持 API 与可视化页面的轻量数据问答应用

不是：
- 通用数据分析 Agent 平台
- 复杂多轮推理型 Copilot
- 可直接用于生产级高风险写操作的数据系统

---

## 2. 从输入到输出的全链路

### 2.1 输入入口
当前有两个主要入口：
- FastAPI 接口：POST /ask、GET /ask
- Streamlit 页面：输入自然语言问题并点击执行

### 2.2 主流程
完整链路如下：

1. 用户输入自然语言问题
2. 进入统一编排入口 ask_question
3. 先做问题分类
   - 判断是简单查询、排名聚合还是复杂分析
   - 判断是否需要 LLM
4. 检索 schema 上下文
   - 优先从 ChromaDB 检索 schema 文档
   - 检索失败时回退到本地 schema / DDL / 字段别名上下文
5. 生成 SQL
   - 简单问题优先走本地规则 / 模板 SQL
   - 复杂问题尝试 LLM
   - 主 LLM 失败时，按 index=2..N 做 fallback provider 级联
   - 全部失败后再回退到本地规则 SQL
6. 执行 SQL 前做安全校验
   - 只允许 SELECT
   - 禁止危险关键字
   - 限制多语句
   - 校验表白名单
   - 自动补 LIMIT 或压缩过大的 LIMIT
7. 执行数据库查询
   - 返回 DataFrame 结果
8. 根据结果自动推荐图表
   - bar / line / pie / scatter / table
9. 输出结果
   - API 返回：SQL、解释、结果预览、chart 配置、耗时、trace 信息
   - UI 返回：表格 + Plotly 图表 + 导出按钮

## 3. 当前已经支持的功能

### 3.1 自然语言查询
已支持中文自然语言转 SQL，适合以下类型：
- 列表查询
- 计数统计
- 平均值 / 最大值 / 最小值
- TopN 排名
- 简单时间条件查询
- 订单、产品、区域、城市等演示数据问题

### 3.2 本地规则优先
当前主链路对常见问题有较强的本地可运行能力：
- YAML 规则匹配
- 模板 SQL 生成
- 本地字段别名映射
- fast fallback 路径

这意味着即使 LLM 不可用，很多常见演示问题仍然能跑通。

### 3.3 LLM 调用与容灾
已支持：
- 通过环境变量配置 1..N 个 provider
- 启动时做健康检查
- 主 provider 失败时自动 fallback
- 将 schema context 提供给模型生成 SQL

### 3.4 Schema 检索
已支持两层 schema context：
- ChromaDB 向量检索
- 本地 DDL / 表结构 / 字段别名 fallback

### 3.5 数据库支持
当前代码层支持：
- SQLite
- MySQL
- PostgreSQL

默认演示库是 SQLite。

### 3.6 查询安全与权限控制
当前已具备基本安全能力：
- 只读模式
- 表白名单
- 危险 SQL 关键字拦截
- 敏感字段识别
- API Key 中间件开关
- 表级权限校验

### 3.7 可视化与导出
当前支持：
- 自动推荐图表类型
- Plotly 图表渲染
- Excel 导出
- PDF 导出

### 3.8 服务接口与运维能力
当前支持：
- 健康检查接口
- schema 查看接口
- metrics 指标接口
- 请求 trace_id 与阶段耗时记录

## 4. 当前能力边界

这是下一阶段开发必须明确的边界。

### 4.1 主要面向演示数据域
当前规则、语义映射、默认 SQL 主要围绕以下表：
- orders
- products
- order_items
- customers

因此它更像“销售演示库上的 Text2SQL”，还不是对任意业务库都稳定泛化的通用系统。

### 4.2 复杂分析能力有限
对于复杂问题，例如：
- 多跳业务语义推理
- 口语化含糊表达
- 多条件复杂组合
- 高度依赖业务知识的分析问法

当前仍然可能：
- 依赖 LLM 质量
- 回退到较保守的 SQL
- 返回默认兜底结果

### 4.3 目前是单轮查询主链路
虽然请求模型里有 session_id 字段，但当前主流程基本仍是单轮问答：
- 没有成熟的上下文记忆编排
- 没有稳定的多轮澄清与追问机制
- 没有真正的分析对话状态管理

### 4.4 当前仅支持只读分析
当前系统明确限制为只读查询：
- 不支持 INSERT / UPDATE / DELETE
- 不适合事务写入、审批流、运维执行类场景

### 4.5 图表能力是自动推荐型，不是完整 BI 设计器
当前图表层可满足演示，但边界也明显：
- 图表类型有限
- 交互配置有限
- 没有复杂 dashboard 编排
- 没有持久化报表能力

### 4.6 导出能力存在环境依赖
PDF 导出当前依赖 Linux 字体路径，对跨平台兼容性还不完全稳健。

### 4.7 LLM 适配层仍偏单一
虽然配置层支持多 provider 概念，但主适配器仍以 OpenAI-compatible 路径为主，通用适配生态还不完整。

---

## 5. 现阶段最适合的使用场景

当前项目最适合：
- 演示 Text2SQL 产品原型
- 做 AI + BI 查询流程展示
- 作为后续工程化重构的基础底座
- 用于验证规则优先 + LLM fallback 的方案

不适合直接承接：
- 生产级复杂 BI 平台
- 高权限数据库操作场景
- 强多租户、强审计、强治理的企业级数据平台

---

## 6. 下一步开发建议

建议下一轮开发优先做以下方向：

1. 提升复杂问题的 SQL 生成稳定性
2. 真正补齐多轮对话与上下文记忆
3. 强化跨库泛化能力与 schema 自适应
4. 完善导出、部署、异常处理与生产化配置
5. 建立更明确的“规则优先 + LLM 补充”治理策略

---

## 7. 一句话总结

当前项目已经具备“从自然语言输入，到安全 SQL 执行，再到结果展示与图表输出”的完整闭环，适合作为 Text2SQL 演示原型与下一步产品化开发的基础版本。
