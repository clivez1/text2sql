# Text2SQL 最终目标总方案

> 项目：text2sql-0412
> 状态：当前生效
> 更新：2026-04-18

---

## 1. 目标重定义

当前项目已经具备演示级 Text2SQL 闭环，但还不是最终目标系统。

本轮方案把项目的终态重新定义为：

1. 一个支持中大型数据库自然语言搜索的成熟轻量级 Text2SQL 项目
2. 一个真正 LLM 优先、支持多轮分析与工具调用的数据分析 Copilot/Agent 平台
3. 一个可治理、可审计、可审批、可回滚的高风险数据操作系统

这里的“轻量级”不是功能弱，而是：

- 单仓可部署
- 模块边界清晰
- 运行面和治理面可控
- 能从云模型平滑切到本地模型
- 先做强治理，再逐步扩能力

---

## 2. 当前基线

当前系统的真实定位见 `10-current-capability-summary.md`。

基线结论：

- 现状是 Demo 和原型底座，不是最终产品形态
- 当前主链路仍是单轮查询，不是多轮 Copilot
- 当前仍是本地规则优先，不是 LLM 优先
- 当前只支持只读分析，不支持受治理写操作
- 当前适合演示库，不适合中大型复杂业务库

因此，本方案不再继续围绕“补几条规则”来推进，而是切换到“最终目标架构先定版，再按阶段落地”的路线。

## 3. 最终目标范围

### 3.1 必须达到的能力

#### A. 中大型数据库自然语言搜索

- 支持 MySQL / PostgreSQL / SQLite，后续可扩更多 SQL 方言
- 支持千表级 schema 检索、候选表列映射和 join path 选择
- 支持业务语义层、指标维度层和 schema 版本化

#### B. LLM 优先主链路

- 问题理解、检索规划、SQL 草拟、解释生成优先由 LLM 完成
- 本地规则降级为 few-shot 资产、修复规则、兜底模板和回归资产
- 支持主模型 + fallback 模型序列
- 支持 OpenAI 协议、Anthropic 协议和本地模型网关三类接入

#### C. 多轮分析 Copilot / Agent

- 支持会话记忆、澄清、追问、分析步骤拆分、工具调用和输出整理
- 支持从“问问题”升级到“完成分析任务”
- 支持分析计划、SQL 执行、结果解释、图表规划和导出在同一会话中协作

#### D. 生产级高风险操作治理

- 默认禁止自由写 SQL
- 只允许受治理的参数化动作和审批式执行
- 必须具备 dry-run、影响预估、审批、事务、审计、快照、回滚和幂等

#### E. 数据可视化与分析表达层

- 不止返回表格，而是返回可解释、可视化、可导出的分析结果
- 图表规划和结果表达进入主链路，而不是只做自动推荐附属能力

---

## 4. 非目标

以下目标不在当前主线范围内：

- 开放式通用浏览器 Agent
- 无治理的任意写操作系统
- 一开始就拆成复杂微服务平台
- 一开始就支持全部异构数据源和全部 BI 场景
- 无约束的 DDL、跨租户写入和不可逆大批量删除

---

## 5. 核心设计原则

1. **LLM 优先，但必须 grounding。** 模型先理解任务，但必须基于 schema、语义层和策略层约束输出。
2. **策略先于执行。** 无论读还是写，都必须经过统一的 policy gate。
3. **工具化而非自由生成。** 多轮 Agent 的能力落在受控工具上，而不是让模型任意输出行为。
4. **协议优先于供应商。** 模型接入按协议抽象，外部云模型只是过渡，本地模型必须可平滑替换。
5. **单仓渐进式演进。** 先在当前仓库内把控制链路做实，再视运行压力拆分控制面和执行面。
6. **先评测，再扩能力。** 每个阶段必须有质量、成本、延迟和安全指标。

---

## 6. 目标架构总览

最终目标系统由五层组成：

1. **入口层**：FastAPI、管理 UI、后续前端工作台
2. **编排层**：会话状态机、任务规划、工具路由、审批流
3. **语义智能层**：LLM Router、Prompt Registry、语义检索、Schema Grounding、SQL Repair
4. **执行治理层**：Policy Gate、Action Catalog、Query Executor、Audit、Rollback
5. **平台支撑层**：Catalog 索引、缓存、评测、监控、配置、模型网关

详细架构见 `11-target-architecture.md`。

## 7. 阶段路线图

### Phase 0：文档重置与基线冻结

目标：

- 归档旧版 Round 文档和阶段方案
- 将 devfile 切换为最终目标方案结构
- 冻结当前边界、未来目标和主从文档关系

退出标准：

- `00/02/03/11-17` 文档体系完整
- 旧方案已做历史快照

### Phase 1：目标架构与接口冻结

目标：

- 冻结目标模块边界
- 冻结核心领域对象：session、catalog、tool call、policy check、audit event、write action
- 冻结配置模型和接口契约

退出标准：

- 代码级目录重构方案明确
- 主要接口和事件对象定义清楚

### Phase 2：LLM 优先链路与模型协议层

目标：

- 从规则优先切到 LLM 优先
- 建立 OpenAI / Anthropic / 本地网关统一 provider 适配层
- 建立任务级模型路由和 fallback 机制
- 引入 SQL draft + validate + repair 闭环

退出标准：

- 简单问题与复杂问题都走统一的 LLM 优先 Router
- 本地规则降级为辅助资产而不是主路由判断器

### Phase 3：中大型数据库语义检索底座

目标：

- 建立 metadata catalog、schema snapshot、join graph、统计信息、业务词汇表
- 建立 BM25 + 向量 + rerank + join path 的混合检索链路
- 让系统从“演示库记忆型”升级到“可泛化检索型”

退出标准：

- 能在中大型库上稳定检索候选表列和 join path
- 关键评测集可稳定达标

### Phase 4：多轮 Copilot / Agent 编排

目标：

- 真正使用 session_id
- 支持澄清、追问、计划、工具调用和会话摘要
- 支持分析任务分步完成

退出标准：

- 单轮查询升级为多轮可追踪任务流
- 会话上下文可裁剪、可恢复、可审计

### Phase 5：可视化与结果表达层

目标：

- 图表规划进入主链路
- 支持采样、聚合、钻取、导出和视图保存
- 让系统输出“分析结果”，而不只是数据表格

退出标准：

- 图表与解释可以稳定复现
- 导出结果包含查询、配置和图表表达

### Phase 6：受治理写操作系统

目标：

- 构建 Action Catalog 和审批流
- 引入 dry-run、影响预估、审批、事务、审计、快照、回滚
- 从只读系统升级为受治理的数据操作系统

退出标准：

- 至少一批受控写动作可以安全上线
- 默认禁止无审批高风险写入

### Phase 7：评测、发布与本地模型化收口

目标：

- 完成统一 eval、SLO、灰度和回归体系
- 将外部云模型接入和本地模型接入收敛到同一协议层
- 形成可发布、可复盘、可审计的版本节奏

退出标准：

- V1/V1.5/V2/V2.5 的验收门槛明确
- 可持续迭代机制形成

## 8. 版本定义

### V1：成熟轻量级只读 Text2SQL

- LLM 优先主链路
- 中大型库语义检索基础能力
- 只读生产化安全与评测

### V1.5：多轮分析 Copilot + 可视化

- 会话记忆
- 澄清与追问
- 图表规划与导出

### V2：受治理写操作

- 参数化动作
- 审批流
- 审计与回滚

### V2.5：本地模型优先企业版

- 本地模型网关进入主模型位
- 云模型保留为 fallback 或特定任务补充

## 11. 当前执行要求

从本文件生效开始，项目主线执行遵循以下顺序：

1. 先定目标架构与领域对象
2. 再重构 LLM 优先链路和 provider 协议层
3. 再做大库检索和语义层
4. 再做多轮 Agent 和可视化
5. 最后解锁写操作治理

不再继续以“补规则、补模板、补演示库问题”为主线推进。
