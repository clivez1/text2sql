## 1. 架构决策参考

### 本地优先原则
（来源：`08-local-first-plan.md`）

核心思路：测试阶段默认 LLM 不可用，主链路必须在本地规则 / 本地 RAG / 本地模板层面具备可运行能力。

推荐链路：
```
用户问题 → 问题分类器（本地规则）
  ↓
A. 高频标准问题 → 规则 SQL
B. 半结构化分析问题 → 本地模板 + schema context
C. 高不确定问题 → 才尝试 LLM
  ↓
SQL 校验 → 查询执行 → 图表推荐
```

### 三层本地增强
（来源：`08-local-first-plan.md`）

**第一层：问题分类** — 判断问题属于哪一类，再决定是否需要 LLM
- 统计类：数量、总数、平均、最大、最小
- 排名类：TopN、最高、前几
- 筛选类：某城市、某区域、某类别
- 列表类：列出、查看、有哪些
- 复杂分析类：趋势、同比、环比、组合条件

**第二层：SQL 模板与规则** — 分类后的问题走本地模板
- 为统计类建立模板
- 为筛选类建立字段词典
- 为时间类建立时间短语映射

**第三层：本地 RAG / Schema Context** — 不依赖 LLM 提升模板选择质量
- 维护字段别名词典
- 维护业务词典
- 检索表说明、字段说明、示例 SQL

## 2. 性能基线参考
（来源：`07-performance-baseline.md`）

**控制面接口基线：**
- `GET /health`：平均 `23.21 ms`
- `GET /schemas`：平均 `8.92 ms`

**主链路基线（优化前）：**
- `/ask` 平均 `28 ~ 36 s`（主要瓶颈在 LLM 失败等待）

**优化后（fast-fallback）：**
- 简单问题从 `33.17 s` 压到 `1.19 s`
- 规则命中已达到"快且更准"

**主要收益来源：** 绕过无效 LLM 调用等待

## 4. 开发规范
（来源：`01-development.md` 精简版）

### 核心设计原则
- 默认只读，严禁写操作
- 优先保证主链路可运行，再增强弹性和可维护性
- LLM 调用失败时必须可降级
- 文档与代码保持对应，避免"计划与现实脱节"

### 文档结构约定
- `devfile/` 是开发主入口
- `docs/` 承担使用说明/API/部署说明角色
- 历史材料放入 `devfile/archive/`
